Since not all of Fieldings constraints were seen as necessary for a particular application, people began referring to [RESTful APIs](https://konghq.com/blog/learning-center/what-is-restful-api)RESTful APIs as a way to distinguish between the theory and implementation of the pattern. Over time, various web frameworks began to bake the principles of RESTful API design into their tooling and made it more straightforward for API developers to build well-designed application interfaces. An API designed according to the principles of REST can be built on any platform. These days, RESTful design revolves around four major design ideas. Let's take a look at each of them. Possibly the most universal aspect of any RESTful API is the decision to make use of HTTP methods for their defined purposes. If you need to create a new resource, POST the representation of your resource to the API. PUT and PATCH should be used to update existing resources either in their entirety or in part. DELETE is the right method, as you may guess, to delete a resource. Another important piece of RESTful design is to make the URL endpoints of your API descriptive of the things they work with.
