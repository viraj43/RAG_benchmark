Horizontal scaling involves adding more independent instances (e.g., virtual machines, containers) to distribute the workload. This approach is highly effective for web applications and microservices architectures, enhancing resilience and availability by eliminating single points of failure. Load balancers intelligently distribute incoming traffic across these multiple instances. Cloud platforms offer robust auto-scaling groups that automatically adjust the number of instances based on predefined performance metrics like CPU utilization or request queue length. While potentially increasing the number of instances, horizontal scaling can be a cost-effective cloud scaling method in the long run by optimizing resource utilization and avoiding the need for expensive, oversized individual servers. A rapidly growing SaaS platform experiencing increasing user sign-ups, for instance, would benefit from automatically adding more application server instances. Key considerations include the need for stateless application design and effective load balancing.
[...]
Vertical scaling involves increasing the resources (CPU, RAM, storage) of an existing instance. It's often a simpler initial approach but has inherent limitations based on the maximum capacity of the underlying hardware. It can also necessitate downtime during upgrades. This approach can lead to higher cloud costs for startups as you opt for more powerful and expensive instance types and is less flexible for fluctuating workloads. A database server experiencing increasing data volume might initially be vertically scaled by increasing its storage capacity and RAM. Key considerations include the limited scalability ceiling, potential downtime during upgrades, and the risk of becoming expensive quickly.

You deploy code in functions, and the provider automatically scales execution based on requests, often leading to cost savings as you only pay for compute time consumed (though costs can rise with sustained high usage).
