# How to build a CI/CD pipeline with GitHub Actions in four simple steps

A quick guide on the advantages of using GitHub Actions as your preferred CI/CD tool—and how to build a CI/CD pipeline with it.

![](https://github.blog/wp-content/uploads/2022/01/build-ci-cd-pipeline-actions_banner.png?resize=1200%2C630)

[Continuous Integration / Continuous Delivery](https://resources.github.com/ci-cd/) (CI/CD) has long been—and continues to be—the domain of [DevOps experts](https://resources.github.com/devops/). But with [the introduction of native CI/CD to GitHub in 2019](https://github.blog/2019-08-08-github-actions-now-supports-ci-cd/) via GitHub Actions, it’s easier than ever to bring CI/CD directly into your workflow right from your repository.

That’s a great thing. As developers, we’re trained to use peer reviews to make sure our code works. But I’m here to tell you we need to disrupt the peer review. If you’re using Git, GitHub, and GitHub Actions to build a CI/CD pipeline, you should have confidence in your code.

I’m going to walk you through exactly how to build your own CI/CD pipeline, right from your repository on GitHub.

*[Find out how GitHub compares to other DevOps and CI/CD platforms >](https://resources.github.com/devops/tools/compare)*

## [Key advantages of using GitHub Actions for CI/CD pipelines](#key-advantages-of-using-github-actions-for-ci-cd-pipelines)

But first, let’s talk through some of the benefits to using GitHub Actions—because let’s be honest, there are a lot of other tools out there. Let me unpack the four big benefits that I’ve come across:

**Respond to any webhook on GitHub**: Since GitHub Actions is fully integrated with GitHub, you can set any webhook as an event trigger for an automation or CI/CD pipeline. This includes things like pull requests, issues, and comments, but it also includes webhooks from any app you have integrated into your GitHub repository. Let’s say you’re going to use any one of the many tools that are out there to run part of your development pipeline. With GitHub Actions, you can trigger CI/CD workflows and pipelines of webhooks from these apps (even something simple, like a chat app message, if you’ve integrated your chat app into your GitHub repository, of course).

**Community-powered, reusable workflows**: You can share your workflows publicly with the wider GitHub community or access pre-built CI/CD workflows in the [GitHub Marketplace](https://github.com/marketplace?category=&query=&type=actions&verification=) (there are more than 11,000 available actions!). Did I mention every action is reusable just by referencing its name? Yes, that too.

**Support for any platform, any language, and any cloud**: GitHub Actions is platform agnostic, language agnostic, and cloud agnostic. That means you can use it with whatever technology you choose.

## [How to build a CI/CD pipeline with GitHub Actions](#how-to-build-a-ci-cd-pipeline-with-github-actions)

Before we dive in, here are a few quick notes:

**GitHub Actions takes a “choose-your-own adventure” type of approach to CI/CD.** You’ll see it the first time you open GitHub Actions in a repository. There are plenty of guided options with pre-built CI workflows you can leverage, per your technology requirements. But you can also build your own CI workflow from scratch if you want to.

**Our example features a website built on Astro and deployed via GitHub Pages.** For both the CI and CD portion of this guide, we’ll be using a website I built and developed called [www.opensauced.pizza](http://www.opensauced.pizza). This website is intended to make it easier for first-time open source contributors to find open source projects to work on and prioritizes projects with clear onboarding flows.

**If you’re a visual learner, we have you covered.** While I’ll walk you through building a simple CI/CD pipeline with my GitHub project [www.opensauced.pizza](http://www.opensauced.pizza), I also have a video you can watch where I put these steps into practice and build a quick CI flow for another project of mine: [hot.opensauced.pizza](https://hot.opensauced.pizza/). This video shows you exactly where to go to start building a streamlined CI pipeline. Give it a watch if you’re a visual learner.

Okay, let’s do this.

### [Step 1: Create or choose a repository, and pick a project](#step-1-create-or-choose-a-repository-and-pick-a-project)

This might sound pretty basic, but the first step to building a CI pipeline with GitHub Actions is creating or choosing a repository on GitHub. You can either use an existing project code base, fork a project you like on GitHub, or start from scratch.

For simplicity’s sake, I’m going to use the [Open Sauced repository](https://github.com/open-sauced/open-sauced) in my Open Sauced project. Feel free to take a closer look by forking this repository and contributing with a fork.

![screenshot of www.opensauced.pizza](https://github.blog/wp-content/uploads/2022/01/open-sauced-repo.png?h=736)

You can see that the Open Sauced repository is relatively simple. The website itself is made with [OneGraph](https://www.onegraph.com/), hosted on [Netlify](https://www.netlify.com/) , and built with HTML, CSS, and JavaScript. I also leverage [Storybook](https://storybook.js.org/) for UI and design work. We’re also using React and npm for package management, installation, and testing—but more on that later.

### [Step 2: Open GitHub Actions in your repository to start building your CI/CD workflow](#step-2-open-github-actions-in-your-repository-to-start-building-your-ci-cd-workflow)

To begin building your CI/CD pipeline, open the GitHub Actions tab in your repository’s top navigation bar.

You should see a list of CI/CD and workflow automation templates that match the technology your project uses (side note: We just improved this entire user flow. [Check out the details later, if you’re interested](https://github.blog/2021-12-17-getting-started-with-github-actions-just-got-easier/)).

![screenshot of workflow options](https://github.blog/wp-content/uploads/2022/01/choose-a-workflow.png?resize=854%2C621)

![screenshot of workflow options](https://github.blog/wp-content/uploads/2022/01/choose-a-workflow.png?resize=854%2C621)

For this project, we’ll leverage a few different CI/CD workflows to test, build, stage, and deploy our code. These include:

A note on YAML

YAML is a markup language that has become a mainstay in declarative automation due to its human-friendly nature as a JSON superset (it uses a lot less brackets, braces, and quotes than other JSON markup language variants). Want to learn more? [Check out this guide](https://learnxinyminutes.com/docs/yaml/).

**A release and build workflow**: This workflow runs tests and enforces lint after releasing code changes to Docker and building the application. It also deploys the final code to our production environment, cuts a release using a similar structure to the automated release notes, bundles the site into a container and publishes to ghcr. From there, it bumps the version number and tag in the repository. This is one of the more complex workflows we’re running, and I’m oversimplifying it somewhat, but [you can see which jobs and steps happen in the workflow for yourself](https://github.com/aaronwinston/open-sauced/blob/main/.github/workflows/release.yml).

**A storybook deployment workflow**: This workflow [deploys any UI component changes to our production website](https://github.com/aaronwinston/open-sauced/blob/main/.github/workflows/storybook.yml) through our frontend UI tech Storybook.

And those are our workflows. The point here is that if you’re working on a solo project or something small, building a CI pipeline doesn’t have to be an intimidating process. You can start with a few simple things (like what’s highlighted above) to make your workflow a little easier.

A CI/CD pipeline can and should be more complex than this if you’re building enterprise software, maintaining a big open source project, or working with a big team on any array of things. But if you’re just getting started, don’t worry about making your CI/CD pipeline meet every single need of a big team or ambitious project—just make it work for you.

You can see that I’ve built in a few workflow automations to simplify the process of maintaining the Open Sauced repository. Case in point: [Compliance workflow](https://github.com/aaronwinston/open-sauced/blob/main/.github/workflows/compliance.yml) (from GitHub user [@mtfoley](https://github.com/mtfoley)), which is triggered whenever someone makes their first pull request (in accordance with our contributor guidelines) on Open Sauced and sends them a message to join our Discord server.

### [Step 3: Make changes to your code to trigger your CI/CD pipeline](#step-3-make-changes-to-your-code-to-trigger-your-ci-cd-pipeline)

For the sake of this exercise, we’re going to make a small change to the website headline (“The path to your next open source contribution”) by adding in the “and more pizza” at the end. The final result will look like this: “The path to your next open source contribution and more pizza.”

![screenshot of website headline for www.opensauced.pizza](https://github.blog/wp-content/uploads/2022/01/website-headline.png?resize=1024%2C426)

![screenshot of website headline for www.opensauced.pizza](https://github.blog/wp-content/uploads/2022/01/website-headline.png?resize=1024%2C426)

Let’s get to work.

Open the src folder, and go to the components subfolder. From there, you’ll want to open the hero.js file. In that file, go to this code:

`<div>
<h1>The path to your next</h1>
<h2>Open Source</h2>
<h1>contribution.</h1>
</div>`

Make the change to the copy by adding in “and more pizza.” It should look like this:

`<h1>The path to your next</h1>
<h2>Open Source</h2>
<h1>contribution and more pizza.</h1>`

### [Step 4: Take a look at the workflow visualizer and live logs to get a full look into how your pipeline is running](#step-4-take-a-look-at-the-workflow-visualizer-and-live-logs-to-get-a-full-look-into-how-your-pipeline-is-running)

Once you push the above change, you get to dive into the fun part: seeing your pipeline work in real time with a workflow visualizer and with live logs.

Okay, maybe this isn’t the most fun part, but knowing how to use both of these tools… you’ll thank me later.

**Let’s go over the workflow visualizer first**. Accessible via the Actions main page, a workflow visualizer can be pulled up by selecting whatever workflow you want to see. You’ll find something like this:

![screenshot of workflow visualizer](https://github.blog/wp-content/uploads/2022/01/workflow-visualizer.png?h=346)

Here, you can see which job in a given workflow happens when—and if they’re working or not with a little green check mark, a yellow sign to show if something’s working, and a red sign to show if a job failed.

This is your YAML workflow, but in visual form, and it makes it easier to see what’s happening when and if it’s working.

**Pro tip:** Since workflow visualization graphics are color-coded to quickly show successful actions, ongoing actions, and actions that failed at a particular step, try using it after you set up a new workflow and trigger it for the first time.

**Now, here’s what you should know about live logs**: First off, they’re your best friend (seriously). Live logs are incredibly useful for figuring out exactly what is working, what’s broken, and why the things you think should be working aren’t working at all. But more to the point, you can go through these logs and look through timestamps, the raw logs, or even download the log itself for local reference.

[Live logs can be accessed directly through the Actions menu](https://docs.github.com/en/actions/monitoring-and-troubleshooting-workflows/using-workflow-run-logs) in your repository and by clicking into any job or workflow. If you did everything right when you built out your CI/CD pipeline, you probably won’t need to look at your live logs.

**But if something goes wrong, these live logs can be very helpful to reference.** Whether you’re looking at timestamps or which part of the process failed, you can determine how to fix the problem.

**Pro tip:** Timestamps can be incredibly useful if you’re debugging a time-sensitive error. By default, a live log will color code which jobs failed and when in a given workflow, and that makes it easier to immediately address the problem.

### [Take this with you](#take-this-with-you)

Whether you’re working on an open source project, a side project, or a work project, there are some big benefits to adopting CI/CD—like producing more consistent and workable releases. But the biggest benefit of all is trusting that your code works after merging it to your main branch, testing it, and deploying it.

With GitHub Actions, building a CI/CD pipeline is a straightforward process and one that lets you focus more extensively on your code instead of all the things that come after it. I’ve been meaning to write an article on GitHub Actions for quite awhile. Having worked with it for several months now, I’ve been very impressed with the power and intuitiveness of it.

This article documents a pipeline I built for my Django music player awhile back. If you’d like to know more about the project, here is a [link](https://medium.com/towards-aws/a-convergence-of-devops-tools-leveraging-django-docker-terraform-and-aws-to-build-a-custom-902733aaed8f) to an overview of it.

## What is GitHub Actions and how is it different from GitHub?

I’m going to assume that if you are reading this article then you are familiar with GitHub. It’s hard to be in the tech world having never heard of it. My assumption is that the vast majority of my readers range from *heard about GitHub and used it a handful of times* to *use it everyday and would be considered experts in it*.

GitHub Actions is the native GitHub CI/CD tool that allows you to automate builds, tests, and deployments from GitHub. The nice thing about it, is that it is very easy to integrate with GitHub repositories (something most of us are probably already using). But GitHub Actions extends beyond the typical DevOps CI/CD pipelines. You can automate all sorts of tasks associated with GitHub; for example adding the appropriate labels whenever an issue is created, closing pull requests or issues based on certain criteria, assigning pull-requests to particular reviewers, generating documentation based on source code. For almost any event that can occur in GitHub, Actions can be used to automate a response to that event.

For the purpose of this article, I will be focusing on the CI/CD functionality of GitHub Actions. Just know that this is barely scratching the surface, and much more can be accomplished with it than this article details.

## Benefits of GitHub Actions

With so many other CI/CD tools out there, why would someone choose GitHub actions?

Here’s a few of the top reasons. Much of these are my personal opinion, but I think a lot of other people would agree:

### Native Solution

If you are already using GitHub for your repositories, the ability to integrate those repositories with a native solution is a big plus. Everything is available to you within that service without having to worry about managing a third-party CI/CD tool.

### Ease of Setup

GitHub Actions was designed with the developer in mind, rather than a DevOps specialist. Getting started is fairly straightforward, and the syntax of the workflow files is incredibly intuitive.

### Cost

GitHub provides a very generous free-tier. For public repositories, or when using self-hosted runners, Actions is always free. For private repositories you are given 2000 free CI/CD minutes per month. If you need more minutes, there are some very affordable premium tier pricing plans (Teams and Enterprise).

### Pre-written Templates

GitHub Actions provides a ton of pre-configured workflow files for almost any type of scenario. Often times, you will only need to slightly modify a file and you can be up and running. These workflow files also provide the added benefit of giving insight on how to best set up your own custom workflow files. To browse them, from your repository go to “Actions”. GitHub will even suggest workflows for you based on the content of the repository.

### Secret Management

You can store your secrets in GitHub securely, which can be used to interact with third-party services, managing necessary keys and passwords.

### Flexibility

In addition to providing some great pre-written solutions out of the box, Actions can be customized to fit almost any CI/CD or automated workflow.

## Prerequisites

For the sake of brevity, some prerequisites are assumed.

* **Git/GitHub:** The basics of using Git and GitHub are assumed and required to follow along. This includes adding files, committing, creating pull requests, branching, etc.
* **Python:** Although not necessary, there are some brief discussions of Python code. An understanding of some basics would be helpful.
* **CI/CD:** This article details a specific tool you can use for CI/CD, but it does not give a broad overview of what CI/CD is, and why it is used.
* **AWS Terminology:** There is some mention of AWS services or resources, such as AMIs, EC2 instances, boto3, access keys, etc. Explanations of them won’t be covered in this article.
* **Docker:** Docker is used extensively within this pipeline. Some basic knowledge is assumed, including Dockerfiles, Compose, Docker Hub, etc.
* **SSH:** Part of the pipeline requires the GitHub runner to SSH into a remote server. It is assumed that you know how to SSH, and know what an SSH key is, host, etc.

## Glossary

Below are some basic terms used by GitHub Actions

* **Runner:** A Runner is the server in which executes a job in a GitHub Actions workflow
* **Workflow:** A workflow is a configurable automated process that is triggered upon a GitHub event.
* **Job:** A job is defined in a workflow file and executes any number of steps. Each job will be executed on its own runner; if many jobs are defined, the default is for them to run concurrently. In the article I will discuss how to make a job dependent upon other jobs.
* **Step:** Steps are the individual stages of each job. Each step within a job will run on the same runner.

## Overview of Pipeline

Press enter or click to view image in full size

![](https://miro.medium.com/v2/resize:fit:1000/1*GKlQad4aLOHYTyXkUqEBJg.png)

This pipeline will first trigger upon a pull-request to the main branch. The basic steps that occur after that are:

1. A staging server (EC2 instance) is started while linting and unit tests run simultaneously.
2. Once the tests have completed, a Docker image is built from a Dockerfile.
3. The Docker image is pushed to Docker Hub
4. The Docker image is pulled from Docker Hub onto the staging server and containers are started with Docker Compose.
5. At this point, I can manually check to make sure the app is working as expected on the staging server.

The next part of the pipeline occurs when the pull request is merged.

1. An Amazon Machine Image is created from the running staging server
2. The staging server is stopped.

The newly created AMI can then be used in production, which is just a matter of changing a variable in the Terraform configuration.

If the pull request is closed without merging, the staging server will be stopped without an AMI being created.

### A few things to point out here:

You’ll notice that the actions “Start Staging Server”, “Run Pylint”, and “Run Unit Tests” occur simultaneously. This is because Action workflows run on separate “runners” for each “job”. Each “job” can contain multiple “steps”. For this reason, each “job” must contain a “step” to checkout the source code.

Furthermore, each workflow must be triggered by a specific event or multiple events, so two workflow files must be created for this pipeline; one for the pull request creation, and one for the pull request being closed. Whether or not the pull request was merged or not determines which jobs will be run.

## Workflow files

The files for workflows must be placed in the *.github/workflows* directory within your GitHub repository. A workflow file can be named anything as long as it has a *.yml* or *.yaml* extension. Additionally, any number of workflow files can be placed within this directory.

### on

The *on* argument specifies what events trigger the workflow. I have chosen *pull\_request*. Again, there are many more available, and a complete list of events can be found [here](https://docs.github.com/en/actions/using-workflows/events-that-trigger-workflows).

Underneath *pull\_requests,* filters can be configured for customizing the trigger event. In my case, I’ve chose to ignore certain paths in the repository, meaning that changes to any of the files in those paths won’t trigger the workflow. I’ve also specified the *branches* to trigger from as *main.* To be clear, main is the target branch, meaning the branch that a pull request is created to merge into, not from.

Another thing to note is that I have not chosen the *push* event to trigger my workflow. This is because I’ve set up a branch protection rule for the main branch that requires pull requests.

### env

Within GitHub, we can store our secret variables so that they don’t have to be hard-coded. The value is kept secret and encrypted by GitHub. Then, to access them we place them in the ${{ }} (for variables) and then *secret.<secret\_name>*.

### pylint

The next job runs Pylint. For those of you that don’t know, a linter is a development tool that checks code for errors and enforces coding standards. It can even check for vulnerabilities. Pylint is one of the more widely used linters for Python, and is highly customizable. The configuration for pylint is stored in a “.pylintrc” file.

After we checkout the code, we install Python 3.11. When using pylint, it is a good idea to stick to a particular version of Python. So even though the GitHub runners already have Python installed on them, it is good practice to include this step. Notice how we use another “composite action” for this step? To specify a particular version of Python we use the *with.python-version* argument.

After the Python installation we install dependencies via a requirements.txt file. For this Django project, there are quite a few dependencies, so pylint, along with all of the other dependencies, are included in the “ci-requirements.txt” file. The reason for naming the requirements file “ci-requirements.txt” is because I also have a separate “docker-requirements.txt” file for building the necessary Docker image.

After the dependencies have been installed, I’ve included a series of commands to run Pylint, checking all of my Python code in the repository.

```
pylint:    runs-on: ubuntu-latest    steps:      - name: Checkout code        uses: actions/checkout@v4      - name: Install Python 3.11        uses: actions/setup-python@v5        with:          python-version: '3.11'      - name: Install Dependencies        run: pip install -r requirements/ci-requirements.txt      - name: Run Pylint        run: |           pylint player --load-plugins pylint_django --django-settings-module=lyria.settings --output-format=text || true          pylint lyria --load-plugins pylint_django --django-settings-module=lyria.settings --output-format=text  || true          pylint ci_scripts --output-format=text || true          pylint bucketActions --output-format=text || truepylint:runs-on:ubuntu-lateststeps: -name: Checkout codeuses:actions/checkout@v4 -name: Install Python3.11uses:actions/setup-python@v5with:python-version:'3.11' -name: Install Dependenciesrun: pip install -rrequirements/ci-requirements.txt -name: Run Pylintrun:|           pylint player --load-plugins pylint_django --django-settings-module=lyria.settings --output-format=text || true          pylint lyria --load-plugins pylint_django --django-settings-module=lyria.settings --output-format=text  || true          pylint ci_scripts --output-format=text || true          pylint bucketActions --output-format=text || true

The unit tests are written into Django in a file within the “player” directory called “tests.py”, which is the default file name for unit tests within Django. In a separate article I may discuss some of the specifics of handling unit tests in Django, but for now just know that the “Run Unit Tests” step runs these tests.

### build\_docker\_image

Notice that the first argument of this job uses the *needs* argument? The *needs* argument is used to create dependencies between jobs; so this job will not run unless the “pylint” and “unit\_tests” jobs have both run and passed. As I explained earlier, the default behavior is for all jobs to run concurrently, which is a great thing since it speeds up the pipeline. To override it, all you have to do is use *needs*.

Moving on..

Again, we checkout the code, then we build the Docker image. There is a “Dockerfile” in the repository, so the *docker build* command builds the image from that, tagging the image with “latest” and the current date.

Next, we login to Docker Hub; again there is a composite action for this. Not all composite actions come from the “actions” organization. Some of them come from the services that they are used for, or from other third parties. This action is a repository in the docker organization. We need to pass in the Docker Hub username and password, which we do with GitHub secrets.

The next step pushes the Docker images to Docker Hub; pretty straightforward.

And finally, we’ve again included a step to stop the staging server in the event of a failure.

**1.)** First we’ll retrieve the staging server’s IP address so we can SSH into it. Once again, this is achieved with a boto3 script. At the end of the “get\_staging\_server\_ip” script we place a block of code like this:

When we print something with a script the output will show up on the GitHub Actions console. So we will see “Lyria Staging Server is now running at: <Public\_IP\_Address>” when our workflow completes this step. The next two lines write the “public\_ip” to the “GITHUB\_OUTPUT” file. This variable can then be used within the current job.

**2.)** In the next step, “Set environment variable for staging server IP”, we set the public\_ip (which was output from the script above) to an environment variable that can be used further down. Notice the syntax:

```
echo "STAGING_SERVER_IP=${{ steps.get_staging_server_ip.outputs.public_ip }}" >> $GITHUB_ENV echo"STAGING_SERVER_IP=${{ steps.get_staging_server_ip.outputs.public_ip }}"${{ steps.get_staging_server_ip.outputs.public_ip } $GITHUB_ENV

When we call a variable outputted from a particular step we need to use *steps.<id\_of\_step>.outputs.<name\_of\_variable>* and then append the environment variable to $GITHUB\_ENV. Now, ${{env.STAGING\_SERVER\_IP}} can be used in subsequent steps.

**3.)** Next, a python script is used to create a “.env” file. The “.env” file is needed to pass environment variables into the containers when the *compose up* command is run. **4.)** In the next step we will SSH into our staging server to stop and remove any containers, images, and Docker networks that currently exist.

To SSH, another composite action is used; *appleboy/ssh-action@master*. The *with* argument is used to pass in the *host* (IP address), *username*, ssh *key*, and the *script* to be run after SSHing.

One thing to point out here is that the first command creates an empty “.env” file. This is needed in order for the *compose down* command to execute properly, since the file is removed after the *compose* *up* command is run. It doesn’t have to have anything in it, but the file does need to be present.

```
- name: Stop Previous Containers and Remove Images        uses: appleboy/ssh-action@master        with:          host: ${{env.STAGING_SERVER_IP}}          username: ubuntu          key: ${{secrets.LYRIA_PRIVATE_KEY}}          script: |            touch .env            sudo docker compose down            sudo docker system prune --all --force -name: Stop Previous Containers and Remove Imagesuses:appleboy/ssh-action@masterwith:host:${{env.STAGING_SERVER_IP}}username: ubuntukey:${{secrets.LYRIA_PRIVATE_KEY}}script:|            touch .env            sudo docker compose down            sudo docker system prune --all --force

Again, the “ssh-action” is utilized for this step.

```
- name: Pull Docker Image and Start Containers on Staging Server        uses: appleboy/ssh-action@master        with:          host: ${{env.STAGING_SERVER_IP}}          username: ubuntu          key: ${{secrets.LYRIA_PRIVATE_KEY}}          script: |            sudo apt update && sudo apt upgrade -y            sudo docker compose up -d            rm .env -name: Pull Docker Image and Start Containers on Staging Serveruses:appleboy/ssh-action@masterwith:host:${{env.STAGING_SERVER_IP}}username: ubuntukey:${{secrets.LYRIA_PRIVATE_KEY}}script:|            sudo apt update && sudo apt upgrade -y            sudo docker compose up -d            rm .env
```

In the compose file, I have specified for the containers to always restart. So when an instance is launched from the AMI that we will build, the containers will start automatically, and there’s no need to have a .env file on the server.

**7.)** At the very end of this job I’ve included the step to stop the staging server in the event of a failure.

The above workflow will be triggered on pull requests. Let’s take a look the next workflow file, which will be triggered on closed pull requests.

## CreateImage.yml

This next workflow is much simpler. Depending on whether the pull request has been merged or not it will either:

* Create an AMI from the running staging server and then stop the server,

or

* Stop the staging server without creating an AMI

At this point, most of the syntax contained in this file has been covered in the previous workflow, but I would like to draw your attention to the *if* arguments. The “create\_image” job contains the following:

```
if: github.event.pull_request.merged == trueif:github.event.pull_request.merged == true
```

So here we are saying “if the github event, pull request merged, is true”. In other words, if the pull request is merged the job will be executed.

In the “stop\_staging\_server” job we have the opposite:

```
if: github.event.pull_request.merged == falseif:github.event.pull_request.merged == false
```

Here we are saying “if the github event, pull request merged, is false”.

So, to recap, the entire workflow will execute on closed pull requests:

```
on:  pull_request:    types: [ closed ]on:pull_request:types: closed
```

Then based on whether or not the pull request is merged or not, different jobs are executed.

### Incrementing Image Tag

Since the scope of this article is to discuss GitHub Actions, and not Python, I will only briefly summarize this point.

The AMI that is created contains a version number in the name tag that is incremented. To achieve this I use the boto3 *ec2.describe\_images* request to obtain a list of the tags. Then the version numbers from the tags are parsed out and the max value of that list chosen (since I may have more than one AMI saved at a given time). That value is then incremented and used in the new name tag with the *ec2.create\_image* request.

## Adding our first workflow

All GitHub repositories come with Actions enabled by default. Browse to your repository, and you’ll see [a tab labeled Actions](https://github.com/jamtur01/animal-farm-nodejs/actions). All you need to do is tell your repository to make use of Actions in your development process.

Actions run [workflows](https://docs.github.com/en/actions/reference/workflow-syntax-for-github-actions), which are usually associated with specific stages in your development process, for example, a workflow that runs when a pull request opens. Inside workflows are jobs—the individual steps in a workflow. For example:

Workflow #1

Clone the repository

Install the prerequisites

Run the tests

These workflows, and the jobs inside them, are defined in YAML-formatted files inside your repository in a directory called **.github/workflows**. Let’s start by adding this directory inside our repo.

## Testing our code with a workflow

Let’s create a workflow to run our tests when we open a new pull request. We create a YAML file to hold this workflow.

*NOTE: The YAML file format can be fiddly to get right. If you want to check if your YAML file is valid, you can use* [*this handy validator*](http://www.yamllint.com/)*.*

And then add our Actions configuration to this file:

You can see our workflow has a descriptive name: **Animal Farm Node.js CI**. We next want to define when our workflow runs. We do this in [the `on` block](https://docs.github.com/en/actions/reference/workflow-syntax-for-github-actions#on). We’ve specified two conditions, both qualified with a specific branch: **main**.

Push - action will trigger if someone pushes to the **main** branch

Pull request - action will trigger if someone opens a pull request from the **main** branch.

You can also configure the **on** block to trigger for other [events](https://docs.github.com/en/actions/reference/events-that-trigger-workflows), such as when a release publishes or on a regular schedule.

Below the **on** block is the **jobs** block, this contains the jobs to be executed by this workflow. We’ve got one job **defined**: **build**. Each job runs on [a specific platform](https://docs.github.com/en/actions/reference/workflow-syntax-for-github-actions#jobsjob_idruns-on). You can currently choose from Linux, Windows, and macOS. In our case, we’re running our job on a container using Ubuntu Linux.

Every job has steps. These are the tasks the job will execute. If one step fails, then generally, the whole job will fail. For our purposes, we want to make sure the last step, which runs our tests, passes before we merge our pull request.

We’ve named each step we’re taking. The first step, called **Checkout repository**, checks out the current repository. In any job, you’ll usually need to do this as the first step, or at least one of the first steps. It makes use of a pre-packaged action: [actions/checkout@v2](https://github.com/actions/checkout). Prepackaged actions are provided by GitHub or community members and usually perform tasks that might otherwise use multiple steps or represent repetitive configuration. Here our **actions/checkout** basically performs a local **git clone** of the repository. You can usually pass options to the pre-packaged action to specify how it’ll execute that task.

Similarly, our second step, called **Use Node.js**, runs another prepackaged action: [actions/setup-node@v1](https://github.com/actions/setup-node). This action takes care of installing Node.js inside the container running our job. We can see one of the action arguments here too. The **with** block tells Actions what Node.js version to install—in our case, the latest version in the **18.x** release.

Our last two steps are at the heart of our job. The next step runs **yarn** to install any Node.js modules we need. The last step executes our tests using **yarn tests**. Let’s see it in action.

We’re going to make a change to our app and then create a pull request for it.

We’ve then edited **app.js** and updated our **bear** to **growl** and added a new animal, a lion who roars, to our **animals** object, committed the result, and pushed it to our repository.

We can then create [a pull request for this branch](https://github.com/jamtur01/animal-farm-nodejs/pull/1). After the request has been created, Actions will initiate our testing workflow. This test is lucky for us because we forgot to update our app’s tests. We can see in our pull request that we have a [failed Actions run](https://github.com/jamtur01/animal-farm-nodejs/pull/1/checks?check_run_id=1967135403) because our tests failed. We can even prevent someone from merging this branch until the tests pass and all our Actions are green.

![A pull request for the roaring lion branch. Shows a green open status with a failed Actions run](//images.ctfassets.net/s5uo95nf6njh/2m6tOC0eOWUpcP5MNe0Oqk/d17dd16e827296fc629ac2f8f60b8079/Inline3_Sotheby-s.png?w=484&fm=png)

Now we can go back and update our test, which will re-run our Github Action, this time resulting in [a successful run](https://github.com/jamtur01/animal-farm-nodejs/pull/1/checks?check_run_id=1967194885).

## A workflow for building Docker images

Finally, we want to update our application’s Docker image and push it to the GitHub Container Registry and the Docker Hub. Let’s create a second workflow to do this.

And populate this file.

We’ve named our new workflow: **Publish Docker image**. This time, our **on** block only triggers when we merge to the **main** branch.  We have one job in our workflow called **build-and-push** that runs on **ubuntu-latest**. Our job checks out our source code using the checkout action we used in our last workflow. It then uses some new pre-packaged actions to set up a Docker build environment using [QEMU](https://github.com/docker/setup-qemu-action) and [Docker buildx](https://github.com/docker/setup-buildx-action).

We use another [Action to login](https://github.com/docker/login-action) to both the GitHub Container Registry and Docker Hub. In these steps, we can see a new construct: variables.

Actions can use variables to customize our workflows and input external data. These can include [secrets](https://docs.github.com/en/actions/reference/encrypted-secrets), like our Docker Hub username and password, or [environment variables](https://docs.github.com/en/actions/reference/environment-variables) like our path, or user-specified values. We’re using a secret, which we can add as a value in our repository settings or, if you’re working within an organization, to the organization. These values are encrypted and open decrypted when being used during our workflow’s execution. We’ve used them to provide credentials to log in to the registries we want to store our Docker image.

*NOTE: Variables also allow us to create* [*“matrix” builds*](https://docs.github.com/en/actions/learn-github-actions/managing-complex-workflows#using-a-build-matrix)*. These builds allow us to run multiple iterations of a workflow using specific values as iterators, for example, running our tests against multiple versions of Node.js or on multiple operating systems.*

Lastly, our workflow builds the new Docker image using the **Dockerfile** from our repository and pushes the built image to both the Docker Hub and GitHub Container Registry.

Let’s merge [our PR](https://github.com/jamtur01/animal-farm-nodejs/pull/1). The merge will trigger two workflows. The [first workflow](https://github.com/jamtur01/animal-farm-nodejs/actions/runs/616754993) runs our tests to confirm that the new **main** branch is working correctly. Our second workflow is [the Docker build and push](https://github.com/jamtur01/animal-farm-nodejs/actions/runs/616754992).

# DevOps Made Simple: A Beginner’s Guide to Setting Up CI/CD Pipelines with GitHub Actions & Kubernetes

## Introduction

Continuous Integration and Continuous Deployment (CI/CD) is a fundamental practice in modern DevOps, ensuring rapid, reliable, and automated software delivery. GitHub Actions and Kubernetes provide a powerful combination for automating CI/CD workflows. This guide will walk you through setting up a CI/CD pipeline using GitHub Actions and deploying an application to Kubernetes step by step.

By the end of this guide, you’ll understand how to:

## Understanding CI/CD, GitHub Actions & Kubernetes

### What is CI/CD?

CI/CD is a software development practice that automates integrating code changes (CI) and delivering software updates (CD). This approach minimizes errors, speeds up releases, and ensures consistency.

### Why GitHub Actions?

GitHub Actions allows developers to automate workflows directly within GitHub repositories. With its simple YAML-based configuration, you can define CI/CD pipelines seamlessly.

### Why Kubernetes?

Kubernetes is an open-source container orchestration platform that automates deployment, scaling, and management of containerized applications.

## Step-by-Step Guide: Setting Up a CI/CD Pipeline

### Step 1: Create a GitHub Repository

Ensure you have a GitHub repository where your application code is stored.

### Step 2: Write a GitHub Actions Workflow

Create a `.github/workflows/deploy.yml` file in your repository with the following structure:

`.github/workflows/deploy.yml`
`name: CI/CD Pipeline
on:
push:
branches:
- main
jobs:
build:
runs-on: ubuntu-latest
steps:
- name: Checkout Code
uses: actions/checkout@v2
- name: Set Up Docker
uses: docker/setup-buildx-action@v1
- name: Build Docker Image
run: docker build -t myapp:latest .
- name: Push Docker Image
run: |
echo "${{ secrets.DOCKER_PASSWORD }}" | docker login -u "${{ secrets.DOCKER_USERNAME }}" --password-stdin
docker tag myapp:latest mydockerhubusername/myapp:latest
docker push mydockerhubusername/myapp:latest
deploy:
needs: build
runs-on: ubuntu-latest
steps:
- name: Deploy to Kubernetes
run: |
echo "${{ secrets.KUBECONFIG }}" | base64 --decode > kubeconfig.yaml
kubectl --kubeconfig=kubeconfig.yaml apply -f k8s/deployment.yaml`

### Step 3: Create Kubernetes Deployment & Service Files

Create a `k8s/deployment.yaml` file:

`k8s/deployment.yaml`
`apiVersion: apps/v1
kind: Deployment
metadata:
name: myapp
spec:
replicas: 2
selector:
matchLabels:
app: myapp
template:
metadata:
labels:
app: myapp
spec:
containers:
- name: myapp
image: mydockerhubusername/myapp:latest
ports:
- containerPort: 80`

### Step 4: Store Secrets Securely

Store your Docker credentials and Kubernetes configuration as GitHub Secrets:

`DOCKER_USERNAME`
`DOCKER_PASSWORD`
`KUBECONFIG`

### Step 5: Push Code and Automate Deployment

Commit and push your code to trigger the GitHub Actions workflow. Your application will be built, pushed to Docker Hub, and deployed to Kubernetes automatically.

# Continuous deployment

You can create custom continuous deployment (CD) workflows directly in your GitHub repository with GitHub Actions.

## In this article

## [About continuous deployment](#about-continuous-deployment)

*Continuous deployment* (CD) is the practice of using automation to publish and deploy software updates. As part of the typical CD process, the code is automatically built and tested before deployment.

Continuous deployment is often coupled with continuous integration. For more information about continuous integration, see [Continuous integration](/en/actions/automating-builds-and-tests/about-continuous-integration).

## [About continuous deployment using GitHub Actions](#about-continuous-deployment-using-github-actions)

You can set up a GitHub Actions workflow to deploy your software product. To verify that your product works as expected, your workflow can build the code in your repository and run your tests before deploying.

You can configure your CD workflow to run when an event occurs (for example, when new code is pushed to the default branch of your repository), on a set schedule, manually, or when an external event occurs using the repository dispatch webhook. For more information about when your workflow can run, see [Events that trigger workflows](/en/actions/using-workflows/events-that-trigger-workflows).

GitHub Actions provides features that give you more control over deployments. For example, you can use environments to require approval for a job to proceed, restrict which branches can trigger a workflow, or limit access to secrets. You can use concurrency to limit your CD pipeline to a maximum of one in-progress deployment and one pending deployment. For more information about these features, see [Deploying with GitHub Actions](/en/actions/deployment/about-deployments/deploying-with-github-actions) and [Managing environments for deployment](/en/actions/deployment/targeting-different-environments/managing-environments-for-deployment).

## [Workflow templates and third-party actions](#workflow-templates-and-third-party-actions)

GitHub offers deployment workflow templates for several popular services, such as Azure Web App. To learn how to get started using a workflow template, see [Using workflow templates](/en/actions/learn-github-actions/using-starter-workflows) or [browse the full list of deployment workflow templates](https://github.com/actions/starter-workflows/tree/main/deployments). You can also check out our more detailed guides for specific deployment workflows, such as [Deploying Node.js to Azure App Service](/en/actions/deployment/deploying-to-your-cloud-provider/deploying-to-azure/deploying-nodejs-to-azure-app-service).

Many service providers also offer actions on GitHub Marketplace for deploying to their service. For the full list, see [GitHub Marketplace](https://github.com/marketplace?category=deployment&type=actions).

## [Next steps](#next-steps)

If your GitHub Actions workflows need to access resources from a cloud provider that supports OpenID Connect (OIDC), you can configure your workflows to authenticate directly to the cloud provider. This will let you stop storing these credentials as long-lived secrets and provide other security benefits. For more information, see [OpenID Connect](/en/actions/deployment/security-hardening-your-deployments/about-security-hardening-with-openid-connect).
