# Quickstart for GitHub Actions

Try out the core features of GitHub Actions in minutes. 

## In this article

## Introduction

GitHub Actions is a continuous integration and continuous delivery (CI/CD) platform that allows you to automate your build, test, and deployment pipeline. You can create workflows that run tests whenever you push a change to your repository, or that deploy merged pull requests to production. 

This quickstart guide shows you how to use the user interface of GitHub to add a workflow that demonstrates some of the essential features of GitHub Actions. 

To get started with preconfigured workflows, browse through the list of templates in the [actions/starter-workflows](https://github.com/actions/starter-workflows)repository. For more information, see [Using workflow templates](/en/actions/writing-workflows/using-starter-workflows). 

For an overview of GitHub Actions workflows, see [Workflows](/en/actions/using-workflows/about-workflows). If you want to learn about the various components that make up GitHub Actions, see [Understanding GitHub Actions](/en/actions/learn-github-actions/understanding-github-actions). 

## Using workflow templates

GitHub provides preconfigured workflow templates that you can use as-is or customize to create your own workflow. GitHub analyzes your code and shows you workflow templates that might be useful for your repository. For example, if your repository contains Node.js code, you'll see suggestions for Node.js projects. 

These workflow templates are designed to help you get up and running quickly, offering a range of configurations such as: 

Use these workflows as a starting place to build your custom workflow or use them as-is. You can browse the full list of workflow templates in the [actions/starter-workflows](https://github.com/actions/starter-workflows)repository.

Committing the workflow file to a branch in your repository triggers the `push`event and runs your workflow. 

If you chose to start a pull request, you can continue and create the pull request, but this is not necessary for the purposes of this quickstart because the commit has still been made to a branch and will trigger the new workflow.

> Pro tip: Since workflow visualization graphics are color-coded to quickly show successful actions, ongoing actions, and actions that failed at a particular step, try using it after you set up a new workflow and trigger it for the first time.

**Now, here’s what you should know about live logs **: First off, they’re your best friend (seriously). Live logs are incredibly useful for figuring out exactly what is working, what’s broken, and why the things you think should be working aren’t working at all. But more to the point, you can go through these logs and look through timestamps, the raw logs, or even download the log itself for local reference. 

[Live logs can be accessed directly through the Actions menu](https://docs.github.com/en/actions/monitoring-and-troubleshooting-workflows/using-workflow-run-logs)in your repository and by clicking into any job or workflow. If you did everything right when you built out your CI/CD pipeline, you probably won’t need to look at your live logs. 

**But if something goes wrong, these live logs can be very helpful to reference. **Whether you’re looking at timestamps or which part of the process failed, you can determine how to fix the problem. 

> Pro tip: Timestamps can be incredibly useful if you’re debugging a time-sensitive error. By default, a live log will color code which jobs failed and when in a given workflow, and that makes it easier to immediately address the problem.

| Create a Workflow File

In your repository, create a folder called .github/workflows/. Inside it, add a file, for example:
.github/workflows/ci.yml
name: CI Pipeline
on:
push:
branches: [ main ]
pull_request:
branches: [ main ]
jobs:
build:
runs-on: ubuntu-latest
steps:
 # Get your code
 - uses: actions/checkout@v3

 # Set up Python
 - name: Set up Python
 uses: actions/setup-python@v4
 with:
 python-version: "3.11"

 # Install dependencies
 - name: Install dependencies
 run: |
 python -m pip install --upgrade pip
 pip install -r requirements.txt

 # Run tests
 - name: Run tests
 run: |
 pytest

This example assumes you’re working with Python, but the structure is similar for other languages.

Add Deployment (CD)

If you want continuous deployment, add another job (or step) that runs only after tests pass. For example, to deploy a static site to GitHub Pages:
deploy:
needs: build
runs-on: ubuntu-latest
steps:
- uses: actions/checkout@v3
- name: Deploy to GitHub Pages
uses: peaceiris/actions-gh-pages@v3
with:
github_token: ${{ secrets.GITHUB_TOKEN }}
publish_dir: ./public
For cloud platforms (Heroku, AWS, Azure, etc.), you’d replace this with their official or community deployment action.

Manage Secrets

If deployment requires secrets (like API keys), store them in your repo’s Settings → Secrets and variables → Actions. Then use them in your workflow as secrets.MY_KEY.

Observe and Debug

Whenever you push, go to the Actions tab in your repo. You’ll see the pipeline running live. If it fails, logs are available for each step.
👉 So, in short:
Put a YAML workflow file in .github/workflows/.
Define when it runs (on:).
Add jobs (build, test, deploy).
Use official actions for setup and deployment. |

| Python doesn't have a specific "decorator attribute" - decorators are functions that take another function/class and return a modified version, applied using @ syntax.
Common use cases: logging, timing, access control, caching, validation, registering functions, and built-ins like @Property, @staticmethod, @classmethod.
Simple example:
def my_decorator(func):
 def wrapper():
 print("Before function")
 func()
 print("After function")
 return wrapper

@my_decorator
def say_hello():
 print("Hello!")

say_hello() # Prints: Before function, Hello!, After function
@Property creates attribute-like access:
class Circle:
 def __init__(self, radius):
 self._radius = radius
 
 @property
 def area(self):
 return 3.14159 * self._radius ** 2

c = Circle(5)
print(c.area) # Access like an attribute, not c.area()
Decorators work on both functions and classes, providing a clean way to modify behavior without changing the original code! |

## Introduction

Continuous Integration and Continuous Deployment (CI/CD) is a fundamental practice in modern DevOps, ensuring rapid, reliable, and automated software delivery. GitHub Actions and Kubernetes provide a powerful combination for automating CI/CD workflows. This guide will walk you through setting up a CI/CD pipeline using GitHub Actions and deploying an application to Kubernetes step by step. 

By the end of this guide, you’ll understand how to: 

## Understanding CI/CD, GitHub Actions & Kubernetes

### What is CI/CD?

CI/CD is a software development practice that automates integrating code changes (CI) and delivering software updates (CD). This approach minimizes errors, speeds up releases, and ensures consistency. 

### Why GitHub Actions?

GitHub Actions allows developers to automate workflows directly within GitHub repositories. With its simple YAML-based configuration, you can define CI/CD pipelines seamlessly. 

### Why Kubernetes?

Kubernetes is an open-source container orchestration platform that automates deployment, scaling, and management of containerized applications. 

## Step-by-Step Guide: Setting Up a CI/CD Pipeline

### Step 1: Create a GitHub Repository

Ensure you have a GitHub repository where your application code is stored. 

### Step 2: Write a GitHub Actions Workflow

Create a `.github/workflows/deploy.yml`file in your repository with the following structure: 

```
name: CI/CD Pipeline

on:
 push:
 branches:
 - main

jobs:
 build:
 runs-on: ubuntu-latest
 steps:
 - name: Checkout Code
 uses: actions/checkout@v2

 - name: Set Up Docker
 uses: docker/setup-buildx-action@v1

 - name: Build Docker Image
 run: docker build -t myapp:latest .

 - name: Push Docker Image
 run: |
 echo "${{ secrets.DOCKER_PASSWORD }}" | docker login -u "${{ secrets.DOCKER_USERNAME }}" --password-stdin
 docker tag myapp:latest mydockerhubusername/myapp:latest
 docker push mydockerhubusername/myapp:latest

 deploy:
 needs: build
 runs-on: ubuntu-latest
 steps:
 - name: Deploy to Kubernetes
 run: |
 echo "${{ secrets.KUBECONFIG }}" | base64 --decode > kubeconfig.yaml
 kubectl --kubeconfig=kubeconfig.yaml apply -f k8s/deployment.yaml
```

### Step 3: Create Kubernetes Deployment & Service Files

Create a `k8s/deployment.yaml`file: 

```
apiVersion: apps/v1
kind: Deployment
metadata:
 name: myapp
spec:
 replicas: 2
 selector:
 matchLabels:
 app: myapp
 template:
 metadata:
 labels:
 app: myapp
 spec:
 containers:
 - name: myapp
 image: mydockerhubusername/myapp:latest
 ports:
 - containerPort: 80
```

## What is GitHub Actions and how is it different from GitHub?

I’m going to assume that if you are reading this article then you are familiar with GitHub. It’s hard to be in the tech world having never heard of it. My assumption is that the vast majority of my readers range from *heard about GitHub and used it a handful of times *to *use it everyday and would be considered experts in it *. 

GitHub Actions is the native GitHub CI/CD tool that allows you to automate builds, tests, and deployments from GitHub. The nice thing about it, is that it is very easy to integrate with GitHub repositories (something most of us are probably already using). But GitHub Actions extends beyond the typical DevOps CI/CD pipelines. You can automate all sorts of tasks associated with GitHub; for example adding the appropriate labels whenever an issue is created, closing pull requests or issues based on certain criteria, assigning pull-requests to particular reviewers, generating documentation based on source code. For almost any event that can occur in GitHub, Actions can be used to automate a response to that event. 

For the purpose of this article, I will be focusing on the CI/CD functionality of GitHub Actions. Just know that this is barely scratching the surface, and much more can be accomplished with it than this article details. 

## Benefits of GitHub Actions

With so many other CI/CD tools out there, why would someone choose GitHub actions? 

Here’s a few of the top reasons. Much of these are my personal opinion, but I think a lot of other people would agree: 

### Native Solution

If you are already using GitHub for your repositories, the ability to integrate those repositories with a native solution is a big plus. Everything is available to you within that service without having to worry about managing a third-party CI/CD tool. 

### Ease of Setup

GitHub Actions was designed with the developer in mind, rather than a DevOps specialist. Getting started is fairly straightforward, and the syntax of the workflow files is incredibly intuitive. 

### Cost

GitHub provides a very generous free-tier. For public repositories, or when using self-hosted runners, Actions is always free. For private repositories you are given 2000 free CI/CD minutes per month. If you need more minutes, there are some very affordable premium tier pricing plans (Teams and Enterprise). 

### Pre-written Templates

GitHub Actions provides a ton of pre-configured workflow files for almost any type of scenario. Often times, you will only need to slightly modify a file and you can be up and running. These workflow files also provide the added benefit of giving insight on how to best set up your own custom workflow files. To browse them, from your repository go to “Actions”. GitHub will even suggest workflows for you based on the content of the repository. 

### Secret Management

You can store your secrets in GitHub securely, which can be used to interact with third-party services, managing necessary keys and passwords. 

### Flexibility

In addition to providing some great pre-written solutions out of the box, Actions can be customized to fit almost any CI/CD or automated workflow.

## Workflow files

The files for workflows must be placed in the *.github/workflows *directory within your GitHub repository. A workflow file can be named anything as long as it has a *.yml *or *.yaml *extension. Additionally, any number of workflow files can be placed within this directory. 

## RunContainers.yml

Let’s take a look at some basic syntax for the workflow file. This is the first workflow for my pipeline: 

For now, let’s focus on some key arguments that are used in this file: 

```
name: # Name of the workflowon: # What events trigger the workflowenv: # Environment variables needed for the workflowpermissions: # Modifies permissions of GITHUB_TOKENjobs: # Top level argument to describe jobs to be performed <name_of_job>: # Name of the job (can be anything) runs_on: # Runner that the job runs on steps: # Top level argument for the steps to be taken - name: # name of the step uses: # Optional - selects an "Action" to use (reusable piece of code) run: # Optional - commands to run
```

There are obviously more arguments in this file, and a complete list of the available syntax can be found [here](https://docs.github.com/en/actions/using-workflows/workflow-syntax-for-github-actions), but in general these are the basic components of this file. Let’s take a look at some individual blocks of this file in more detail. 

We’ll start with the first three top level arguments; *name, on, *and *env *. 

```

```

The *name *argument is self explanatory, so let’s move on to the *on *argument: 

### on

The *on *argument specifies what events trigger the workflow. I have chosen *pull_request *. Again, there are many more available, and a complete list of events can be found [here](https://docs.github.com/en/actions/using-workflows/events-that-trigger-workflows). 

Underneath *pull_requests, *filters can be configured for customizing the trigger event. In my case, I’ve chose to ignore certain paths in the repository, meaning that changes to any of the files in those paths won’t trigger the workflow. I’ve also specified the *branches *to trigger from as *main. *To be clear, main is the target branch, meaning the branch that a pull request is created to merge into, not from. 

Another thing to note is that I have not chosen the *push *event to trigger my workflow. This is because I’ve set up a branch protection rule for the main branch that requires pull requests. 

### env

Next, we have the environment variables ( *env *). If you look closely, some of these variables are hardcoded, but others have the following syntax: 

```
${{secrets.<NAME_OF_VARIABLE>}}
```

Within GitHub, we can store our secret variables so that they don’t have to be hard-coded. The value is kept secret and encrypted by GitHub. Then, to access them we place them in the ${{ }} (for variables) and then *secret.<secret_name> *. 

I’ll briefly explain how to create a secret in GitHub at the end of this section. For now, let’s move on 

### jobs

Let’s take a look at the first couple of *jobs *. 

What we have below are three *jobs *: 

The first thing after the name of the job is the *runs_on *argument. This is the container or machine that the job will be executed on. The GitHub hosted runners are Docker images based on either Linux, Mac or Windows. I’ve chosen to use ubuntu, which is actually the only Linux distribution available. You’re also free to choose any kind of self-hosted runner. A ‘large runner’ with additional memory and compute power can be selected for an additional cost. 

Next, *steps *need to be included to execute within each job. Notice how each job contains the “Checkout code” step? Then below, the *uses *argument is called with the argument “actions/checkout@v4”. This is an example of a “composite action”. Essentially, it is reusable code that can be called within a single step of a job. There is a whole list of these “actions” in the [actions](https://github.com/actions)organization on GitHub. When you call the checkout action, your code will automatically be checked out to the GitHub runner. 

```
jobs: start_staging_server: runs-on: ubuntu-latest steps: - name: Checkout code uses: actions/checkout@v4 - name: Install boto3 run: pip install boto3 - name: Start Staging Server id: start_staging_server working-directory: .github/ci_scripts run: python start_staging_server.py pylint: runs-on: ubuntu-latest steps: - name: Checkout code uses: actions/checkout@v4 - name: Install Python 3.11 uses: actions/setup-python@v5 with: python-version: '3.11' - name: Install Dependencies run: pip install -r requirements/ci-requirements.txt - name: Run Pylint run: | pylint player --load-plugins pylint_django --django-settings-module=lyria.settings --output-format=text || true pylint lyria --load-plugins pylint_django --django-settings-module=lyria.settings --output-format=text || true pylint ci_scripts --output-format=text || true pylint bucketActions --output-format=text || true unit_tests: runs-on: ubuntu-latest steps: - name: Checkout code uses: actions/checkout@v4 - name: Install Python 3.11 uses: actions/setup-python@v5 with: python-version: '3.11' - name: Install Dependencies run: pip install -r requirements/ci-requirements.txt - name: Run Unit Tests run: python manage.py test player - name: Stop Staging Server if: ${{ failure()}} working-directory: .github/ci_scripts run: python stop_staging_server.py
```

### start_staging_server

Take a look at the rest of the code under the “start_staging_server” job. The next step installs boto3. Whenever shell commands need to be run we use the *run *argument. 

In the last step of the “start_staging_server” job a boto3 script is run that starts an EC2 instance (staging server). We can specify a working directory, and then use a shell command to run the script. The *id *argument is just a unique identifier for that step. 

```
start_staging_server: runs-on: ubuntu-latest steps: - name: Checkout code uses: actions/checkout@v4 - name: Install boto3 run: pip install boto3 - name: Start Staging Server id: start_staging_server working-directory: .github/ci_scripts run: python start_staging_server.py
```

It’s also important to note how to call the workflow environment variables in your python scripts. 

You’ll need to import the “os” module: 

```
import os
```

And then use the “environ” method from “os” to call the environment variables from the workflow file: 

```
access_key_id = os.environ.get('AWS_ACCESS_KEY_ID')secret_access_key = os.environ.get('AWS_SECRET_ACCESS_KEY')instance_id = os.environ.get('STAGING_INSTANCE_ID')
```

In the above example we set three variables. The red text (or blue if you’re in dark mode) are the name of the variables in the workflow file. Now you are free to use the variables *access_key_id *, *secret_access_key *, and *instance_id *as you normally would in the script. 

### pylint

The next job runs Pylint. For those of you that don’t know, a linter is a development tool that checks code for errors and enforces coding standards. It can even check for vulnerabilities. Pylint is one of the more widely used linters for Python, and is highly customizable. The configuration for pylint is stored in a “.pylintrc” file. 

After we checkout the code, we install Python 3.11. When using pylint, it is a good idea to stick to a particular version of Python. So even though the GitHub runners already have Python installed on them, it is good practice to include this step. Notice how we use another “composite action” for this step? To specify a particular version of Python we use the *with.python-version *argument. 

After the Python installation we install dependencies via a requirements.txt file. For this Django project, there are quite a few dependencies, so pylint, along with all of the other dependencies, are included in the “ci-requirements.txt” file. The reason for naming the requirements file “ci-requirements.txt” is because I also have a separate “docker-requirements.txt” file for building the necessary Docker image. 

After the dependencies have been installed, I’ve included a series of commands to run Pylint, checking all of my Python code in the repository. 

```
pylint: runs-on: ubuntu-latest steps: - name: Checkout code uses: actions/checkout@v4 - name: Install Python 3.11 uses: actions/setup-python@v5 with: python-version: '3.11' - name: Install Dependencies run: pip install -r requirements/ci-requirements.txt - name: Run Pylint run: | pylint player --load-plugins pylint_django --django-settings-module=lyria.settings --output-format=text || true pylint lyria --load-plugins pylint_django --django-settings-module=lyria.settings --output-format=text || true pylint ci_scripts --output-format=text || true pylint bucketActions --output-format=text || true
```

### unit_tests

For this job we also need to install Python 3.11, and the necessary dependencies from the “ci-requirements.txt” file. After that, we provide a command to run the unit tests. The unit tests are written into Django in a file within the “player” directory called “tests.py”, which is the default file name for unit tests within Django. Django uses Pytest, a popular unit testing tool for Python, with an additional plugin to handle some Django specific cases. In a separate article I may discuss some of the specifics of handling unit tests in Django, but for now just know that the “Run Unit Tests” step runs these tests. 

Take a look at the next step. The “Stop staging server” step only runs if the previous step fails; if the unit tests fail we want the staging server to stop, since, if that is the case, we will not be building an AMI. To do so, I’ve included an *if *argument and passed in the *failure *variable. Again, a boto3 script is used to stop the staging server in the “.github/ci_scripts” directory. By default, no other step will complete if the previous one fails, so without the *if: ${{ failure }} *variable we’d be left with a running EC2 instance. 

```
unit_tests: runs-on: ubuntu-latest steps: - name: Checkout code uses: actions/checkout@v4 - name: Install Python 3.11 uses: actions/setup-python@v5 with: python-version: '3.11' - name: Install Dependencies run: pip install -r requirements/ci-requirements.txt - name: Run Unit Tests run: python manage.py test player - name: Stop Staging Server if: ${{ failure()}} working-directory: .github/ci_scripts run: python stop_staging_server.py
```

### build_docker_image

Notice that the first argument of this job uses the *needs *argument? The *needs *argument is used to create dependencies between jobs; so this job will not run unless the “pylint” and “unit_tests” jobs have both run and passed. As I explained earlier, the default behavior is for all jobs to run concurrently, which is a great thing since it speeds up the pipeline. To override it, all you have to do is use *needs *. 

Moving on.. 

Again, we checkout the code, then we build the Docker image. There is a “Dockerfile” in the repository, so the *docker build *command builds the image from that, tagging the image with “latest” and the current date. 

Next, we login to Docker Hub; again there is a composite action for this. Not all composite actions come from the “actions” organization. Some of them come from the services that they are used for, or from other third parties. This action is a repository in the docker organization. We need to pass in the Docker Hub username and password, which we do with GitHub secrets. 

The next step pushes the Docker images to Docker Hub; pretty straightforward. 

And finally, we’ve again included a step to stop the staging server in the event of a failure. 

```
build_docker_image: needs: [pylint, unit_tests] runs-on: ubuntu-latest steps: - name: Checkout code uses: actions/checkout@v4 - name: Build Lyria Docker image run: | DATE=$(date +'%Y-%m-%d') docker build --tag krisaff84/lyria:${DATE} \ --tag krisaff84/lyria:latest . - name: Log in to Docker Hub uses: docker/login-action@v3 with: username: ${{secrets.DOCKER_USERNAME}} password: ${{secrets.DOCKER_PASSWORD}} - name: Push image to Docker Hub run: docker push krisaff84/lyria:$(date +'%Y-%m-%d') && docker push krisaff84/lyria:latest - name: Stop Staging Server if: ${{ failure()}} working-directory: .github/ci_scripts run: | pip install boto3 python stop_staging_server.py
```

### pull_image_and_start _containers

The *needs *argument is used again to make sure that the “build_docker_image” and “start_staging_server” jobs have both run without failure.

**1.) **First we’ll retrieve the staging server’s IP address so we can SSH into it. Once again, this is achieved with a boto3 script. At the end of the “get_staging_server_ip” script we place a block of code like this:


**4.) **In the next step we will SSH into our staging server to stop and remove any containers, images, and Docker networks that currently exist. 

To SSH, another composite action is used; *appleboy/ssh-action@master *. The *with *argument is used to pass in the *host *(IP address), *username *, ssh *key *, and the *script *to be run after SSHing.


**5.) **After we’ve removed all of the previous Docker components, the new necessary files need to be copied to the staging server. We use another composite action by “appleboy” called “scp-action”. 

Within the *with *argument, we once again pass in the *host *, *username *, and *key *. If your files are not in the root directory you can add a path to them as well.


**6.) **Finally, we actually SSH into our instance and pass in the *docker compose up *command. After the Docker command is run, the “.env” file is removed, since it contains sensitive information. 

Again, the “ssh-action” is utilized for this step. 

```
- name: Pull Docker Image and Start Containers on Staging Server uses: appleboy/ssh-action@master with: host: ${{env.STAGING_SERVER_IP}} username: ubuntu key: ${{secrets.LYRIA_PRIVATE_KEY}} script: | sudo apt update && sudo apt upgrade -y sudo docker compose up -d rm .env
```

In the compose file, I have specified for the containers to always restart. So when an instance is launched from the AMI that we will build, the containers will start automatically, and there’s no need to have a .env file on the server.

### Incrementing Image Tag

Since the scope of this article is to discuss GitHub Actions, and not Python, I will only briefly summarize this point. 

The AMI that is created contains a version number in the name tag that is incremented. To achieve this I use the boto3 *ec2.describe_images *request to obtain a list of the tags. Then the version numbers from the tags are parsed out and the max value of that list chosen (since I may have more than one AMI saved at a given time). That value is then incremented and used in the new name tag with the *ec2.create_image *request.

## Failing Checks

I’m going to change some code to make one of the unit tests fail intentionally. What we should see is that the status checks will fail and we are not allowed to merge. 

### Require Status Checks

Before we proceed with this test I’m going to edit my branch protection rules to require status checks. To do so, go to “Settings” from the repository, then select “Branches”. 

Press enter or click to view image in full size 

Next you’ll select your branch (main), and click edit. Scroll to where it says “Require status checks to pass before merging”. Select that box and then select previous status checks in the search field. 

You also need to scroll down and check the box “Do not allow bypassing the above settings”. 

Press enter or click to view image in full size 

Once that’s done, save the changes. 

** EDIT: After I published this article I realized that checking the “Do not allow bypassing …” box would not work in all situations. What happens, is if you make changes to any of the ignored paths, the workflow does not run, but the status checks are still required, so GitHub won’t allow you to merge any commits that only change one of those paths. I’m working on a more elegant solution, but for now you can simply leave this box unchecked to merge changes in situations where the workflow does not run.
